# Prior Art Benchmark: Conversation Memory & Context Management for AI Agents

*A cited, web-research-only benchmark for an engineering/research team building a self-hosted AI agent platform. No claims are made about any specific proprietary or internal codebase; all findings and recommendations are at the level of public architecture and pattern.*

## TL;DR
- The field has converged on a layered pattern — verbatim recent transcript + just-in-time compaction + external persistent memory retrieved on demand — and for a system that already has a strong semantic knowledge graph plus a hard "no silent context loss" invariant, the two highest-leverage, **non-duplicative** additions are: (1) an episodic **"threads to pull"** layer that returns lightweight POINTERS to real past turns (modeled on Zep/Graphiti's non-lossy episode subgraph and TraceMem's thread IDs), and (2) a structured, **user-editable per-conversation state object** (goals/decisions/open threads/artifacts) that subsumes the prose rolling summary and becomes the compaction target (modeled on Letta memory blocks, Claude Code's `claude-progress.txt`, and Cline's Focus Chain).
- Compression triggers cluster into three families: **threshold-triggered** (Claude Code/Cline compact near context-window fill; LangChain `ConversationSummaryBufferMemory` at a token limit), **cadence-triggered background formation** (Letta sleep-time every N steps; LangMem background; Zep/Graphiti at each ingestion event), and **hot-path/agent-decided** (Anthropic memory tool, LangMem hot-path). Zep and Mem0 report ~90% latency and token reductions versus full-context, but retrieval-based memory degrades sharply on multi-fact and implicit recall — the classic episodic gap.
- Editable/persistent memory is proven UX but carries documented failure modes: **staleness**, **concurrent-write conflicts**, and **memory poisoning via indirect prompt injection** (Unit 42, MINJA, AgentPoison, MemoryGraft). Provenance, retrieval gating, bounded blocks, and single-writer ownership are the mitigations the literature supports.

## Key Findings

1. **Raw-until-compaction is the dominant coding-agent pattern, and its own authors say it is not sufficient alone.** Anthropic's compaction "distills the contents of a context window" and reinitializes a new window with the summary plus "the five most recently accessed files"; but their long-running-agents work shows compaction "doesn't always pass perfectly clear instructions to the next agent" and had to be supplemented with structured external files and even full context resets. This validates moving the compaction target from free prose to a structured state object.

2. **The "no silent context loss" invariant is a genuine differentiator.** Most compaction systems make irreversible retain/discard decisions. Anthropic explicitly notes "it is difficult to know which tokens the future turns will need," and their newest "Managed Agents" design stores context as a durable object outside the window (an append-only session log, queryable via `getEvents()`). Zep/Graphiti stores raw episodes "non-lossily." The commissioning invariant — no turn leaves view unless covered by the summary — is essentially a correctness contract these systems approximate but rarely guarantee.

3. **A strong semantic knowledge graph already covers the "facts" job; the gap is episodic recall.** Cognitive science (Tulving) and CoALA both separate semantic (decontextualized facts) from episodic (specific experienced events with context). Benchmarks show retrieval-based memory (Mem0) collapses on multi-evidence and implicit recall — down to 25% accuracy with 6 evidence items vs ~83% for long-context (Salesforce Research's ConvoMem). This is precisely the "weak at episodic" symptom in the commissioning context.

4. **Pointers-not-payloads is a real, published pattern.** Anthropic's "just in time" context uses "lightweight identifiers (file paths, stored queries, web links)" loaded on demand. Zep/Graphiti maintains bidirectional indices so "semantic artifacts can be traced to their sources for citation or quotation." TraceMem retrieves lightweight "thread entries each comprising a title, summary, and unique thread ID" before fetching full content. This directly models the desired "threads to pull."

5. **Editable memory blocks are proven UX, with named concurrency and poisoning pitfalls.** Letta memory blocks are labeled, size-limited, persisted, and agent-editable; Letta explicitly warns that "Multiple agents doing memory_rethink on the same block simultaneously leads to lost updates" and prescribes a single "owner" for heavy edits. Security research (Unit 42, MINJA, AgentPoison, MemoryGraft) demonstrates durable behavioral compromise via poisoned memory.

## Details

### SECTION A — Long-conversation memory architectures

#### A.1 Raw-until-compaction (Anthropic Claude Code; Cline; Aider)

**Anthropic Claude Code / Claude Developer Platform.** Compaction is "taking a conversation nearing the context window limit, summarizing its contents, and reinitiating a new context window with the summary." In Claude Code, "we implement this by passing the message history to the model to summarize and compress the most critical details. The model preserves architectural decisions, unresolved bugs, and implementation details while discarding redundant tool outputs or messages. The agent can then continue with this compressed context plus the five most recently accessed files."
- **Kept verbatim vs compressed:** recent transcript stays raw; older turns are summarized. The lightest-touch form is tool-result clearing — "once a tool has been called deep in the message history, why would the agent need to see the raw result again?" — shipped as `clear_tool_uses_20250919`, triggered on an `input_tokens` value and keeping the N most recent tool uses.
- **When triggered:** threshold-based as the conversation approaches the window limit.
- **Silent-loss risk:** "overly aggressive compaction can result in the loss of subtle but critical context whose importance only becomes apparent later." Guidance: "Start by maximizing recall … then iterate to improve precision." The "Effective harnesses for long-running agents" post shows compaction alone left features "half-implemented and undocumented," motivating structured artifacts (`claude-progress.txt`, a JSON feature list of 200+ testable features) and, for Sonnet 4.5, full context resets to counter "context anxiety" (a tendency to wrap up prematurely as the limit nears; Opus 4.5 "largely removed that behavior on its own"). The "Managed Agents" work stores context durably in an append-only session log accessible via `getEvents()`, explicitly because "irreversible decisions to selectively retain or discard context can lead to failures … It is difficult to know which tokens the future turns will need."
- **Reported numbers (Anthropic self-report):** on a 100-turn web-search task, context editing + memory tool gave **84% token reduction and a 39% performance improvement** (29% for context editing alone).

**Cline.** "Auto Compact": near the window limit, Cline "creates a comprehensive summary … preserves all technical details, code changes, and decisions … replaces the conversation history with the summary … continues exactly where it left off." It explicitly notes the prior silent-loss behavior: "Previously, Cline would truncate older messages when hitting context limits, losing important context." Checkpoints enable rollback ("you never truly lose context"). Focus Chain reinjects a todo list on a cadence (default every 6 messages). Community threads document erratic firing (too late → "request too long," then too early at ~140k after a fix), illustrating threshold fragility.

**Aider.** A repo map built by a graph-ranking algorithm sends "just the most relevant portions" within a token budget (a pointer-like approach); `/clear` resets history. Users note the whole history is otherwise included with only a basic AI summary — motivating finer control.

#### A.2 Continuous rolling summary (LangChain)

**`ConversationSummaryBufferMemory`.** "It keeps a buffer of recent interactions in memory, but rather than just completely flushing old interactions it compiles them into a summary and uses both … it uses token length rather than number of interactions to determine when to flush." A `moving_summary_buffer` holds the running summary; when the buffer exceeds `max_token_limit`, oldest messages are pruned into the summary.
- **When triggered:** on `save_context` when token count exceeds `max_token_limit`.
- **Tradeoffs:** documented failure modes include the summary buffer growing past the limit over time (langchain #17888) and `prune()` not respecting the limit (langchainjs #4452); "Potential Information Loss: Important details may be omitted during summarization." This is the exact double-compression / summary-drift risk to avoid. LangChain's newer stack replaces these with middleware (`SummarizationMiddleware`, context-editing/tool-pruning).

#### A.3 Paged / virtual context (MemGPT → Letta)

**MemGPT (Packer et al., 2023, "MemGPT: Towards LLMs as Operating Systems").** "Virtual context management," an OS-inspired hierarchy: main context (RAM-like fixed window) and external context (disk-like), split into recall storage (full event history) and archival storage (vector-indexed read/write). The LLM self-edits via function calls (`core_memory_append`, `archival_memory_search`, …) driven by "explicit instructions within the preprompt." On overflow it uses recursive summarization: "it generates a new summary from the old summary plus evicted messages."
- **Kept verbatim vs compressed:** main context (system prompt + recent messages + editable core-memory blocks) verbatim; overflow summarized and paged to searchable external stores.
- **Tradeoff:** recursive summarization "compounds errors over time."

**Letta.**
- **Memory blocks (core memory):** "a labeled section of the context window with an associated character limit," persisted with a `block_id`, editable by agent (memory tools) or developer (API), shareable, and `read_only`-lockable. "Always visible — no retrieval needed," prepended in XML-like format. Origin: MemGPT human + persona blocks.
- **Sleep-time compute:** a background sleep-time agent shares blocks with the primary agent and "can modify the memory asynchronously," turning "raw context" into "learned context." Triggered every N steps (default 5). The primary agent lacks core-memory-edit tools; those belong to the sleep-time agent (clean separation). Letta recommends a stronger/slower model for sleep-time (Sonnet/gpt-4.1) and a fast one for the primary. On a SWE-repair scenario, sleep-time compute "used 3,000 fewer tokens on average, while producing better repair plans."
- **Concurrency:** named directly — single owner + append-only `memory_insert`; MemFS uses git worktrees so subagents "write memory concurrently without blocking the main agent."

#### A.4 Temporal knowledge graph (Zep/Graphiti; Mem0; LangMem)

**Zep / Graphiti (Rasmussen, Paliychuk, Beauvais, Ryan, Chalef, 2025, arXiv:2501.13956).** A temporally-aware KG with three hierarchical subgraphs: **episode** (raw messages/text/JSON — "Episodes serve as a non-lossy data store from which semantic entities and relations are extracted"), **semantic entity** (extracted entities + fact edges), **community** (label-propagation clusters with iterative map-reduce summaries).
- **Kept verbatim vs compressed:** episodes non-lossy; entities/facts extracted; communities summarized. "Episodes and their derived semantic edges maintain bidirectional indices … semantic artifacts can be traced to their sources for citation or quotation, while episodes can quickly retrieve their relevant entities and facts."
- **When triggered:** at ingestion, per episode; entity extraction uses the current message plus the last n=4 messages for context.
- **Retrieval:** three search functions — cosine semantic similarity (φ_cos), Okapi BM25 (φ_bm25), and breadth-first graph search (φ_bfs, which "can accept nodes as parameters … using recent episodes as seeds") — combined then reranked. Rerankers: Reciprocal Rank Fusion, MMR, a graph **episode-mentions** reranker ("frequently referenced information becomes more readily accessible"), a **node-distance** reranker, and **cross-encoders** ("highest computational cost"). Retrieval "avoids any LLM calls."
- **Temporal model:** bi-temporal — timeline T (valid time) and T′ (transaction time); four timestamps (`created_at`, `expired_at`, `valid_at`, `invalid_at`). New edges can invalidate old ones via LLM contradiction detection; facts are invalidated, not deleted (e.g., "Kendra loves Adidas" → invalidated when she switches to Puma). Bulk ingestion skips invalidation.
- **Numbers (vendor self-report):** DMR 94.8% vs MemGPT 93.4%; on LongMemEval, **up to +18.5% accuracy with gpt-4o (+15.2% with gpt-4o-mini) and ~90% lower latency vs full-context**, average context tokens dropping **115,000 → ~1,600** ("less than 2% of the baseline tokens"). Sub-results: temporal reasoning +38.4%, multi-session +30.7%, preference +184%. Zep reports a **300ms P95 retrieval latency** (vendor benchmark).
- **Tradeoffs:** the paper's experiments retrieved only the 20 most relevant edges (facts) + entity summaries — the **atomized-fact path**, not raw episodic recall — mirroring the commissioning "returns facts, never the original thread" gap, even though episode nodes are architecturally retrievable. Ingestion is LLM-heavy (multiple calls per episode), so write latency ≫ read latency. **Independent caution:** a 2026 third-party evaluation (arXiv:2601.07978) reported Graphiti at only 56.03% accuracy on LoCoMo vs 77–81% for Mem0/RAG — directionally opposite to Zep's own numbers — so treat vendor benchmarks as directional, not settled.

**Mem0 (Chhikara et al., 2025, arXiv:2504.19413).** Two-phase: extract salient facts into a memory bank, then retrieve top-K per query.
- **Numbers (vendor self-report):** on LOCOMO, +26% relative LLM-as-judge score over OpenAI memory (66.9% vs 52.9%); p95 latency 1.44s vs 17.12s full-context (91% lower); ~90% token reduction (~1.8K vs ~26K). Full-context still had the highest accuracy (~73% J).
- **Independent (Salesforce Research, ConvoMem, arXiv:2511.10523, 75,336 QA pairs):** "long context memory maintains 70–82% accuracy … compared to 30–45% for state-of-the-art RAG-based memory systems like Mem0, when operating on conversation histories under 150 interactions"; with 6 evidence items Mem0 drops to ~25% (a 58-point gap). Full-context wins on accuracy for roughly the first 150 conversations; Mem0 wins decisively on cost/latency beyond that.
- **Tradeoff:** strong latency/cost, weak on multi-fact and implicit reasoning — the semantic-vs-episodic gap.

**LangMem (LangChain SDK).** Implements CoALA's trichotomy: semantic (facts/profiles), episodic (past interactions as few-shot examples), procedural (behavior/prompt rules). Two formation modes: **hot path** (agent calls a memory tool mid-conversation — adds latency) and **background/subconscious** (async extraction — no latency hit, needs trigger logic). Consolidation merges related memories and resolves contradictions to prevent "memory hoarding." Gating is explicit: "Some memories (core procedural memory) may be data-independent … always present … Some are data-dependent and may be recalled based on semantic similarity."

### SECTION B — Episodic vs semantic memory in agent systems

**Cognitive-science distinction.** Tulving's split — episodic memory "binds context to its memory content, such as when, where, and why an event was encountered," enabling retrieval by contextual cues, vs semantic memory as decontextualized facts. CoALA (Sumers, Yao, Narasimhan, Griffiths, 2023, arXiv:2309.02427) ports this to LLM agents: working memory + long-term episodic (experiences), semantic (world facts), procedural (skills/code). A position paper (arXiv:2502.06975) argues "episodic memory is the missing piece for long-term LLM agents."

**Who does episodic retrieval well, and how they avoid overload.**
- **Generative Agents (Park et al., 2023, UIST '23, arXiv:2304.03442) — the reference design.** A memory stream ranked by a weighted sum of three min-max-normalized signals (equal weights, α=1): **recency** (exponential decay, factor 0.995, "over the number of sandbox game hours since the memory was last retrieved"), **importance** (LLM-rated 1–10), and **relevance** (cosine similarity to the query). Only "top-ranked memories that fit in the context window are included." **Reflection** synthesizes higher-level inferences into a reflection tree (observations as leaves, abstract inferences as non-leaf nodes), and — quoting the paper — "we generate reflections when the sum of the importance scores for the latest events perceived by the agents exceeds a threshold (150 in our implementation). In practice, our agents reflected roughly two or three times a day." This salience + recency-decay + reflection-hierarchy stack is copied across many later systems.
- **Retrieval gating.** LangMem frames "whether to retrieve at all"; Anthropic's agentic search invokes retrieval as a tool only when the model recognizes need.
- **Pointer/lead pattern (what surfaces vs stays retrievable).** Anthropic "just in time": maintain "lightweight identifiers … and use these references to dynamically load data into context at runtime." Zep/Graphiti's bidirectional episode indices trace facts back to source turns. **TraceMem** (arXiv:2602.09712) agentic search: episodic retrieval → memory-card selection → thread extraction, where each thread entry is "a title, summary, and unique thread ID," and full content is fetched "directly from the vector database" only for selected threads. Classic cognitive architectures (Soar) stored "the event sequence as a list of pointers." This is the strongest prior art for "threads to pull."

### SECTION C — Editable / user-visible memory blocks

- **Letta / MemGPT memory blocks.** Labeled, described, size-limited, persisted, agent- and developer-editable, shareable; `read_only` locks a block. "Always visible — no retrieval needed," prepended in XML-like format. "The description is the main information used by the agent to determine how to read and write to that block."
- **Anthropic memory tool + memory files.** A client-side `/memories` directory Claude can create/read/update/delete across sessions; "the model only issues read/write instructions — the actual storage lives on your side." System prompt auto-injects "IMPORTANT: ALWAYS VIEW YOUR MEMORY DIRECTORY BEFORE DOING ANYTHING ELSE." Recommended caps: files ≤64KB, directory ≤10MB/user, per-user root keyed by `user_id`. Claude Code's `CLAUDE.md` is a hierarchical, version-controllable persistent-instruction file loaded up front. Anthropic is testing **Memory Files** (multiple topic-organized documents) to replace the single-running-file "Classic" consumer memory, explicitly to escape the context-window ceiling of one monolithic file.
- **Cursor rules / `.cursorrules`.** Persistent instructions injected every interaction; modern `.cursor/rules/*.mdc` split by concern with four activation modes (Always, Auto-Attached by glob, Agent-Requested, Manual). Documented pitfalls: rules can "eat 25%" of context, and recency bias means a rule "can fire correctly at the start of a conversation and still get ignored five messages later" (workaround: reinforcement). Community "Memory Bank" adds `activeContext.md`/`progress.txt`.
- **OpenAI ChatGPT Memory.** Two mechanisms: **saved memories** (explicit, user-manageable, always in context — "work similarly to custom instructions, except our models update them automatically") and **reference chat history** (automatic, background-curated via "dreaming"). Staleness is acknowledged ("Traditional memory systems can become stale"; the "you're still in Singapore" example). OpenAI stores extracted facts, not verbatim transcripts. Security researchers show saved memories are placed in a "Model Set Context" system-prompt section and can be written via indirect prompt injection.
- **"Agent writes to a structured state doc" patterns.** Claude Code `claude-progress.txt` + JSON feature list; Cline Focus Chain / Memory Bank; the widely-cited practitioner pattern `prd.json` + `progress.txt` + `AGENTS.md` ("state lives outside the agent's context"). Letta **MemFS** is a git-backed memory filesystem with version history and conflict resolution.

**Pitfalls (explicit).**
- **Staleness:** OpenAI's Singapore example; Cursor rules going stale; Claude "fading memory" as files grow.
- **Edit conflicts / concurrent writes:** Letta — "Multiple agents doing memory_rethink on the same block simultaneously leads to lost updates"; mitigation is single owner + append-only inserts, or git worktrees (MemFS).
- **User trust / transparency:** file-based memory (CLAUDE.md, memory tool) prioritizes "transparency and control"; ChatGPT's automatic chat-history learning is criticized because "you don't know what it learned or when it's using it." Claude is "designed to be transparent about when it is using memory."
- **Prompt injection / memory poisoning:** Palo Alto **Unit 42** showed indirect prompt injection via a malicious webpage can corrupt Amazon Bedrock Agent long-term memory, planting instructions that "influence completely unrelated sessions." **MINJA** (Dong et al., 2025) injects malicious records "via ordinary queries." **AgentPoison** (Chen et al., 2024) backdoors memory/knowledge base. **MemoryGraft** implants "malicious successful experiences" for "durable, trigger-free behavioral drift." Persistence is the core danger: once injected, the instruction "remains in all subsequent contexts." **Agent Security Bench** (Zhang et al., 2024, arXiv:2410.02644) reports a highest average attack success rate of **84.30%** (mixed attacks 84.3%, direct prompt injection 72.7%, PoT backdoors 42.1%; note memory-poisoning specifically was only 7.9% — the risk is real but attack-dependent). OWASP LLM Top 10 (2025) ranks prompt injection first.

### SECTION D — Prompt / tool patterns for effective memory use

**Anthropic's published guidance (concrete phrasings to adopt).**
- Guiding principle: find "the smallest possible set of high-signal tokens that maximize the likelihood of some desired outcome."
- System prompts at "the right altitude" — between brittle hardcoded logic and vague guidance; organize into sections (`<background_information>`, `<instructions>`, `## Tool guidance`) with XML/Markdown headers.
- Memory-tool protocol (verbatim): "IMPORTANT: ALWAYS VIEW YOUR MEMORY DIRECTORY BEFORE DOING ANYTHING ELSE. MEMORY PROTOCOL: 1. Use the `view` command … to check for earlier progress. … ASSUME INTERRUPTION: Your context window might be reset at any moment, so you risk losing any progress that is not recorded in your memory directory." Hygiene: "always try to keep its content up-to-date, coherent and organized. You can rename or delete files that are no longer relevant. Do not create new files unless necessary."
- Compaction heuristic: preserve "architectural decisions, unresolved bugs, and implementation details while discarding redundant tool outputs"; "maximize recall … then improve precision."
- Note-taking: "Like Claude Code creating a to-do list, or your custom agent maintaining a NOTES.md file."
- Tool design: "Keep your three to five most-used tools always loaded"; avoid bloated tool sets — "If a human engineer can't definitively say which tool should be used … an AI agent can't be expected to do better."

**Others.**
- **LangMem:** gate recall by data-independent (always present) vs data-dependent (recall by similarity/time/context); prefer background formation to avoid hot-path latency; consolidate to prevent hoarding.
- **Letta:** describe blocks well (descriptions drive read/write behavior); assign a single owner for heavy edits.
- **OpenAI:** memory is for "high-level preferences and details, not exact templates or large blocks of verbatim text."
- **Cursor practitioners:** "Write dense, not wordy"; examples beat prose ("a 10-line example is often more effective than 30 lines"); reinforce a rule in the next prompt when drift appears.

### SECTION E — Synthesis for the specific design constraints

Given a hard "no silent context loss" invariant AND an already-strong semantic knowledge graph, the highest-leverage, non-duplicative additions are:

1. **Episodic "threads to pull" as pointers, layered on the existing graph's raw-turn store.** The graph already stores raw turns; the gap is that retrieval returns only atomized facts. Add an episodic index that returns lightweight **leads** — `{thread_id, title, one-line summary, timestamp, salience}` — modeled on TraceMem thread entries and Zep's bidirectional episode indices. The agent chooses whether to "pull" the full thread via a second tool call. Gate retrieval (LangMem) and rank leads by Generative-Agents-style recency×importance×relevance (with an optional cross-encoder rerank, à la Zep) so you surface a few leads, not a firehose. This fills the episodic gap without rebuilding the semantic graph.

2. **A structured, user-editable per-conversation state object as the compaction target.** Replace the free-prose rolling summary with a big-block state doc (goals / decisions / open threads / artifacts), modeled on Letta memory blocks + Claude Code `claude-progress.txt` + Cline Focus Chain. This is strictly better than prose because it (a) is human-inspectable and editable, (b) resists drift by having named slots, and (c) gives the "no silent loss" invariant a concrete surface: each dropped turn must be reflected in a slot before eviction. Keep blocks bounded (Letta character limits; Anthropic file-size caps) and version them (MemFS git-backed).

3. **Prompt patterns that make memory tools reliable.** Adopt Anthropic's "ASSUME INTERRUPTION" framing, LangMem's recall-gating heuristic, and Letta's block-description discipline as system-prompt scaffolding.

**Why these are non-duplicative:** the semantic KG already does facts/entities/reflexes well (Zep/Mem0/LangMem semantic territory); the additions target the two things graphs are documented to do poorly — **episodic recall of specific threads** and **per-conversation working-state fidelity**.

## Recommendations

**Stage 1 (now): make the invariant enforceable.** Convert the rolling prose summary into a structured state object (goals/decisions/open threads/artifacts) and make it the compaction target. Enforce the contract mechanically: a turn may be evicted only after a just-in-time write confirms its content is represented in a slot (Anthropic "maximize recall first"). *Benchmark:* zero eviction events without a corresponding state write in a trace audit.

**Stage 2: add the episodic pointer layer.** Return leads (thread_id + title + summary + timestamp + salience), never full threads, from episodic retrieval; require a second explicit "pull" tool call to load a full thread. Rank leads by recency×importance×relevance (Generative Agents) with an optional cross-encoder rerank (Zep). Gate retrieval so the agent decides whether to recall at all (LangMem). *Benchmark:* on LongMemEval multi-session and temporal-reasoning slices, target closing the ~30% long-context accuracy drop without inflating context beyond a fixed lead budget (e.g., ≤5 leads, ≤2 pulled threads/turn).

**Stage 3: harden editable memory.** Attach provenance to every block/fact (source turn, author = user vs agent, timestamp), enforce single-writer ownership for heavy edits (Letta) or worktree-style isolation (MemFS), bound block sizes, and treat any externally-ingested content as untrusted (Unit 42/MINJA). *Benchmark:* run an Agent Security Bench-style memory-poisoning red-team; require that injected instructions from tool/web content never alter behavior in a later unrelated turn.

**Thresholds that change the plan:** If episodic leads routinely overload context (agent pulls too many threads), tighten the lead budget and raise the gating bar before adding capacity. If the structured state object drifts or bloats, cap slot sizes and add a background "sleep-time" consolidator (Letta) on the summarizer tier rather than compacting the summary again (avoid double-compression). If write latency from graph ingestion or background formation hurts UX, move formation fully off the hot path (LangMem background; Letta sleep-time cadence).

## What NOT to do (anti-patterns)

- **Do NOT dump full past threads into context.** Reintroduces context rot (Anthropic) and defeats the pointer pattern; return leads and pull on demand (TraceMem, Anthropic just-in-time).
- **Do NOT double-compress.** Summarizing an already-summarized buffer compounds errors (MemGPT recursive summarization "compounds errors over time"; LangChain summary-drift bugs). Compact the raw transcript into the structured state object once; don't re-summarize the summary.
- **Do NOT let editable blocks grow unbounded.** Monolithic files cause "fading memory" (Claude) and eat context (Cursor rules "eat 25%"); enforce size caps and topic-split (Anthropic Memory Files).
- **Do NOT trust editable/persistent memory without provenance.** Indirect prompt injection poisons long-term memory durably (Unit 42, MINJA, AgentPoison, MemoryGraft); tag source and author, and treat ingested content as untrusted.
- **Do NOT retrieve without gating.** Always-on retrieval wastes context and adds noise; decide whether to recall at all (LangMem), and rank/decay/importance-filter what you surface (Generative Agents).
- **Do NOT allow concurrent uncontrolled writes to a shared block.** "Multiple agents doing memory_rethink on the same block simultaneously leads to lost updates" (Letta); designate a single owner or use append-only inserts / git worktrees.
- **Do NOT make irreversible retain/discard decisions with no recoverability.** "It is difficult to know which tokens the future turns will need" (Anthropic); keep raw turns durably stored (Zep non-lossy episodes; Anthropic session log) so the invariant holds even if the summary missed something.
- **Do NOT rely on threshold-only compaction tuning.** Cline's erratic firing (too late → "request too long," then too early at ~140k) shows brittle thresholds; pair thresholds with structured state and rollback/checkpoints.

## Caveats

- **Vendor benchmark numbers are largely self-reported and sometimes contradicted.** Mem0's 91% latency / 90% token reductions and Zep's +18.5% accuracy / 90% latency / 300ms P95 come from the vendors' own papers/blogs; Salesforce Research's independent ConvoMem shows full-context still beats Mem0 on accuracy for the first ~150 conversations, and a 2026 third-party eval (arXiv:2601.07978) reported Graphiti at 56.03% vs 77–81% for Mem0/RAG on LoCoMo — directionally opposite to Zep's own results. Treat single-source numbers as directional.
- **Memory-poisoning attack success is attack-dependent.** ASB's headline 84.30% is for mixed attacks; memory-poisoning specifically scored 7.9% in that benchmark. The threat is real and persistent, but success rates vary widely by vector and defense.
- **Several "successor" details are recent and moving fast.** Anthropic "Memory Files" is described as in testing (not GA); Letta Constellation/MemFS and Claude "Managed Agents" are new. Product specifics may change.
- **Some sources are third-party explainers.** I prioritized primary papers and official docs and flagged where a claim rests on a partner blog (e.g., Neo4j's 300ms P95) or vendor self-report. ChatGPT memory internals ("dreaming," "Model Set Context" injection) are partly reverse-engineered by security researchers — treat implementation specifics as informed inference, not official spec.
- This report makes no claims about any specific proprietary/internal codebase; all recommendations are at the architecture/pattern level.
