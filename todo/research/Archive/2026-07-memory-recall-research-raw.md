# AgentX Memory System: Inventory and a State-of-the-Art Research Foundation for Accurate Memory Dispatch and Stable Long-Term Growth

## TL;DR
- **AgentX already implements a genuinely advanced, multi-store cognitive memory architecture** — four memory types (working/episodic/semantic/procedural) over Neo4j + pgvector + Redis, a five-technique RecallLayer (hybrid BM25+vector with RRF, entity-centric graph traversal, query expansion, HyDE, self-query), Generative-Agents-style reranking (salience/temporal/access boosts), background consolidation, salience decay, contradiction/correction handling, and a shipped Context Ledger. The single highest-leverage gap for **retrieval accuracy** is the absence of a cross-encoder reranking stage and an adaptive per-query dispatcher; for **stable growth** it is the reliance on flat exponential salience decay plus hard deletion rather than a bitemporal, tombstone-based, consolidation-hierarchy model.
- **The literature converges on three fixes AgentX is already partly architected for:** (1) two-stage retrieval (broad hybrid recall → cross-encoder rerank) plus adaptive routing/gating (Adaptive-RAG, Self-RAG) for accuracy; (2) hierarchical summarization (RAPTOR), graph community summaries, and bitemporal validity + reinforcement-modulated decay (Zep/Graphiti, MemoryBank, Bjork's storage/retrieval-strength distinction) for stable growth; and (3) event-sourced write-ahead logging + durable checkpoints for continuation. AgentX's own `Memory-Roadmap.md` independently specifies most of these (PPR/HippoRAG §3.6, community summaries §3.7, bitemporal facts §1.7, contextual bandit over recall §3.5, event-sourced log §1.1), so this report largely validates and prioritizes that roadmap against the literature rather than proposing an orthogonal direction.
- **One contested-but-important finding should shape AgentX's core design bet:** a January 2026 preprint (arXiv 2601.00821) reports evidence that memory-representation *fidelity to the source* strongly drives retrieval accuracy — i.e., LLM extraction that departs too far from raw text loses to near-verbatim storage. AgentX is an extraction-first system, so it should (a) retain verbatim turns as first-class retrievable units alongside extracted facts (its `conversation_logs` already stores them with embeddings — nearly free), (b) treat extracted facts as an *augmentation*, never a *replacement*, for the raw episodic store, and (c) preserve surface detail (especially date strings) during extraction. **Note: this source is a single-author, non-peer-reviewed preprint that appears to have changed its conclusion/title across versions — treat its exact figures as unverified (see Caveats).**

## Key Findings

1. **AgentX's stack is fixed and modern (Python ≥3.14, Django 5.2):** Neo4j (`neo4j>=6.0.3` driver; docs cite server 5.15+, graph + native vector indexes, 1024-dim cosine), PostgreSQL 16 + pgvector 0.7+ (IVFFlat lists=100, BRIN time-series, partitioned audit log), Redis 7 (working memory, TTL), `sentence-transformers` + `faiss-cpu` + `torch` + `transformers` (local embeddings, default `bge-m3` @ 1024-dim or OpenAI `text-embedding-3-small`), `networkx` (in-process graph algorithms), `tiktoken` (token budgeting), and provider SDKs (OpenAI, Anthropic). Any recommendation must be compatible with this stack — and notably, **every advanced technique surveyed below is implementable within it** (pgvector for hybrid, Neo4j GDS or `networkx` for PPR/Leiden, a local cross-encoder via `sentence-transformers`).
2. **Memory types map cleanly onto the CoALA taxonomy** (working / episodic / semantic / procedural), the field-standard cognitive-architecture decomposition (Sumers et al., 2023). AgentX additionally implements per-agent self-knowledge channels (`_self_{agent_id}`) and a scope hierarchy (`_global → user → project → _self → conversation`) more sophisticated than most production systems.
3. **Retrieval is already multi-strategy** but uses static toggles and a linear rerank; it lacks (a) a cross-encoder second stage (the single largest documented accuracy lever in RAG), (b) learned/adaptive per-query technique selection, and (c) multi-hop graph ranking (Personalized PageRank).
4. **Stable growth mechanisms exist but are first-generation:** flat `×0.95^days` salience decay (≈14-day half-life), threshold-based promotion, `dedupe_entities`, and a `cleanup` job that **hard-deletes** low-salience entities. The literature (MemoryBank, FadeMem, Bjork) favors reinforcement-modulated decay, tombstones over deletion, and a summarization hierarchy so forgetting compresses rather than destroys.
5. **Continuation is partially solved:** Redis-backed working memory + a shipped Context Ledger (`agent/context_ledger.py`) with priority-based token allocation and a maintained "salient core" block. The roadmap correctly identifies that Redis `allkeys-lru` eviction currently threatens durability of resumable state (§1.6) and that an event-sourced write path (§1.1) is needed for lossless resume.

## Details

### Part A — Inventory of AgentX's current memory techniques (the research foundation)

**A.1 Technology stack** (from `pyproject.toml`, referenced deps, and the docs "Technology Stack" table). AgentX v0.21.147. Three coordinated stores behind one `AgentMemory` interface:

| Store | Version | Role | Indexes |
|---|---|---|---|
| Neo4j | driver `>=6.0.3` (docs cite server 5.15+) | knowledge graph; semantic, episodic-graph, procedural, entities | 4 native vector indexes (Turn/Entity/Fact/Strategy embeddings, 1024-dim cosine); channel + composite B-tree indexes |
| PostgreSQL + pgvector | 16+ / 0.7+ | `conversation_logs` (raw turns + embeddings), `memory_timeline`, `tool_invocations`, `user_profiles`, partitioned `memory_audit_log` | IVFFlat (lists=100) on embeddings; BRIN on timestamps |
| Redis | 7+ | working memory, session state, consolidation locks/jobs, checkpoints | key patterns `working:{user}:{channel}:{conv}:*`; `allkeys-lru` @ 512MB |

Embeddings: `sentence-transformers` (local `bge-m3`) or OpenAI `text-embedding-3-small`; all vectors 1024-dim and **derived, never canonical** (exports are text-only, re-embedded on import). Extraction default model: `claude-3-5-haiku-latest` (configurable via `extraction_model`). ORM/migrations via SQLAlchemy + Alembic; Neo4j schema versioned on a `_SchemaMeta` node.

**A.2 Memory types** (from the "Memory Capabilities" manifest and "Memory Architecture" docs).
- **Working** (`memory/working.py`, Redis): recent turns, active goal, chain-of-thought scratchpad, context values with TTL (default 1h). Scoped per user/channel/conversation.
- **Episodic** (`memory/episodic.py`, Neo4j graph + pgvector): every `Turn` stored as a node with `embedding vector(1024)`, plus a PostgreSQL `conversation_logs` mirror. Retrieval = vector similarity + temporal filtering/recency boost. `(:Conversation)-[:HAS_TURN]->(:Turn)-[:FOLLOWED_BY]->(:Turn)`.
- **Semantic** (`memory/semantic.py`, Neo4j): `Entity` and `Fact` nodes with typed relationships (`RELATED_TO`, `WORKS_FOR`, `ABOUT`, `DERIVED_FROM`, `PROMOTED_FROM`, `SUPERSEDES`). Facts carry `confidence`, `source`, `source_turn_id`, `salience`, `access_count`. Entity salience + decay. Multi-hop traversal limited to 2–3 hops.
- **Procedural** (`memory/procedural.py`, Neo4j): tool-usage trajectories, learned `Strategy` nodes (success/failure counts → success rate), a distillation loop (`stage_procedure_candidate` → `distill_procedures` → `learn_procedure`), and a **reflex core** (`get_reflex_procedures`) that injects top-strength procedures into every prompt.
- **Goals** (semantic, conversation-scoped): `add_goal`/`complete_goal`, created by `TaskPlanner`; `SUBGOAL_OF`/`BLOCKED_BY` edges.

**A.3 Recall/dispatch (the `RecallLayer`, `memory/recall.py`).** Base `MemoryRetriever` fuses vector similarity, graph traversal, temporal filtering, and a diversity rerank with fixed weights (Episodic 0.3, Facts 0.25, Entities 0.2, Procedural 0.15, Recency 0.1). The `RecallLayer` wraps it with **five toggleable techniques**: (1) Hybrid BM25+vector with **Reciprocal Rank Fusion**; (2) Entity-centric graph traversal from matched entities; (3) Query expansion (question→statement); (4) **HyDE** (hypothetical document embedding); (5) Self-query (LLM extracts structured filters: time/entity-type/channel). Techniques run over the channel set `[active, _self_{agent_id}, _global]`, merge+dedupe, then **rerank by salience + temporal boost (current=1.2×, past=0.7×) + access boosts**. A "stable salient core" (`get_salient_core`) injects top-salience facts via a cheap non-vector `ORDER BY salience` as a prio-70 Context-Ledger block, with query-driven recall riding along as a droppable prio-30 supplement. Per-technique latency/counts tracked by `RecallMetrics`. Settings via `GET/POST /api/memory/recall-settings`.

**A.4 Consolidation & lifecycle (background `ConsolidationWorker`).** Jobs: `consolidate_episodic_to_semantic` (15 min), `detect_patterns` (60 min), `apply_memory_decay` (daily), `cleanup_old_memories` (daily), `promote_prominent_facts` (60 min), `manage_audit_partitions` (daily). Pipeline: relevance pre-filter (cut ~75% of extraction calls) → combined single-call extraction (entities+facts+relationships) → embedding-based entity linking → 3-layer verification (claim-hash → semantic-duplicate → entity-scoped → LLM `check_contradictions`) → store/supersede. Confidence calibration is static (explicit=0.95, implied=0.85, inferred=0.70, uncertain=0.50). Decay: `decayed = initial × 0.95^days_since_access` (≈14-day half-life). Promotion thresholds: confidence ≥ 0.85, access_count ≥ 5, conversations ≥ 2. Corrections detected by "actually…"/"no, I meant…" patterns → `[:SUPERSEDES]`.

**A.5 Continuation.** Working memory + a shipped **Context Ledger** (v0.21.90): each contributor registers a `LedgerBlock(priority, min_tokens, max_tokens, shrink_fn, mandatory)`; the allocator shrinks then drops under token pressure (`shrink_tail`, `shrink_lines_newest_n`, `shrink_memory_to_facts`). Checkpoints/plan snapshots/rolling summary live in Redis (currently under `allkeys-lru` eviction risk). Context gating: `ToolOutputCompressor` + `tool_output_chunker` + Focus-style trajectory compression. Full export/import (round-trippable JSON, stable UUIDs, text-only).

**A.6 What the roadmap already plans** (`Memory-Roadmap.md`, reconciled to v0.21.88): event-sourced `conversation_logs` as write-ahead log (§1.1), bitemporal facts `valid_from/valid_to` + `asserted_at/superseded_at` (§1.7), spaced-repetition decay with `stability`/`last_reinforced` (§2.4), unified tombstones (§2.5), `eval_recall` golden set (§2.7), flash facts / two-speed memory (§3.2), surprise-gated encoding (§3.3), anticipatory prefetch (§3.4), **contextual bandit over recall techniques** (§3.5), **Personalized PageRank / HippoRAG-style** (§3.6), **Leiden graph community summaries** (§3.7), reflexion scoring (§3.8), generative replay / "sleep-time compute" (§3.9), and a degradation ladder L0–L3 (§3.11).

### Part B — Advanced techniques from the literature, mapped onto AgentX

#### B.1 Retrieval accuracy & relevancy (Priority 1)

**Foundational scoring — Generative Agents (Park et al., UIST 2023).** Retrieval score = normalized sum of **recency** (exponential decay, factor 0.995/hour since last access), **importance** (LLM-rated 1–10 at encoding), and **relevance** (cosine similarity to a query memory), min-max scaled and (in the original) equally weighted. AgentX already implements analogues (salience≈importance, temporal boost≈recency, vector score≈relevance) but with **hand-set weights** and importance-from-confidence rather than a dedicated importance rating. *Mapping:* AgentX's reranker is a Generative-Agents reranker missing an explicit LLM importance signal at encode time; adding one (cheap, at consolidation) would improve which facts enter the salient core. The literature critique ("Learning What to Remember," arXiv 2606.12945) is that these weights should be *learned*, not fixed — exactly AgentX roadmap §3.5's contextual bandit.

**Two-stage retrieval with cross-encoder reranking — the single largest documented lever.** Across independent 2026 benchmarks, adding a cross-encoder reranker over hybrid-fused candidates yields the largest single accuracy gain: e.g., +17.2 pp MRR@3 and +12.1 pp Recall@5 over unreranked hybrid on a text-and-table benchmark (arXiv 2604.01733), with MRR@3 jumping 0.433→0.605. The industry-standard pattern is **retrieve top-~100 via hybrid RRF → rerank to top-3–10 via a cross-encoder** (e.g., a `bge-reranker`, runnable locally via `sentence-transformers`, already in AgentX's deps). *Gap:* AgentX's rerank stage is a metadata/salience linear boost, **not** a query-document cross-encoder. This is the highest-ROI accuracy addition and needs no new infrastructure. The roadmap references a "backlog cross-encoder reranker" but it is not yet a shipped RecallLayer stage.

**Adaptive routing / gating — dispatch the right strategy per situation.** Adaptive-RAG (Jeong et al., NAACL 2024) trains a T5 classifier to route queries into no-retrieval / single-step / multi-step tiers by complexity, matching always-expensive baselines at lower cost. Self-RAG (Asai et al., ICLR 2024) uses reflection tokens to decide *when* to retrieve and to self-critique relevance. FLARE, Probing-RAG, SKR, and CRAG are variants (retrieval triggered by generation confidence, hidden-state probes, self-knowledge, or post-hoc correction; Probing-RAG skips retrieval in 57.5% of cases while gaining 6–8 points). *Mapping:* AgentX's five techniques run on static toggles; a router (rule-based first, then the roadmap §3.5 Thompson-sampling bandit keyed on query shape + channel kind + intent class) would spend expensive techniques (HyDE, self-query, cross-encoder) only where they pay. Reward signal = roadmap §2.2's "usage signal" (fact survived ledger allocation / was cited). This directly serves "memory dispatched accurately per situation."

**Hybrid retrieval + RRF.** RRF scores each doc `1/(k+rank)` summed across rankers (k=60 the Cormack et al. default; lower k emphasizes top ranks), ranking by position not raw score so BM25 and dense scores need no normalization. AgentX already does this — correctly built. The refinements: tune k and consider convex combination (α=0.5 balanced fusion sometimes beats RRF), and **gate the candidate pool to ≥50–100 before reranking** (reranking is ineffective at 20 candidates: Recall@5 0.458 → 0.826 at 50 → 0.888 at 100).

**Multi-hop graph ranking — HippoRAG / Personalized PageRank.** HippoRAG (Gutiérrez et al., NeurIPS 2024, arXiv 2405.14831) builds a schemaless KG via LLM OpenIE, then at query time seeds **Personalized PageRank** from query entities and ranks passages by aggregated stationary probability, adding **node specificity** (a graph-native IDF: inverse passage-frequency weighting). Its abstract reports it "outperforms the state-of-the-art methods remarkably, by up to 20%"; concretely it reaches **89.1% Recall@5 on 2WikiMultiHopQA vs 68.2% for ColBERTv2**, with single-step retrieval **10–20× cheaper and 6–13× faster** than iterative IRCoT. It solves the "path-finding problem": retrieving facts connected only through entity chains sharing no lexical overlap with the query. *Mapping:* AgentX already has the entity graph (`ABOUT`, `RELATES_TO`, `SUPERSEDES`) and can run PPR via Neo4j GDS or `networkx` (both available). Roadmap §3.6 specifies exactly this — the strongest upgrade to the existing entity-centric technique and the natural primitive for a future agentic `deep_recall`.

**Query rewriting / expansion / metadata filtering.** LongMemEval (Wu et al., ICLR 2025) shows **time-aware query expansion** (LLM parses a time range and restricts scope) yields +7–11% on temporal reasoning, and **fact-augmented key expansion** + **session decomposition** improve indexing (Recall@k +4%, end-to-end QA +5%). AgentX's self-query technique already extracts time/entity filters; the additional lever is *session/round-level granularity* for episodic storage and *chain-of-note* reading (extract supporting evidence per retrieved memory before answering).

**⚠️ The representation bet — verbatim vs. extracted (a decision, not a tweak).** A January 2026 single-author preprint (Tao An, arXiv 2601.00821) argues that memory-representation **fidelity to source text is the dominant driver of retrieval accuracy** in long conversations — extraction that "generalizes away surface detail" (especially date strings) loses to near-verbatim storage. The reconciling thesis across the paper and its peers: *retrieval accuracy tracks how far the representation departs from the source.* **This source is contested and its exact numbers should not be relied upon:** the version located under DOI-adjacent lookups is titled "CogCanvas: Verbatim-Grounded Artifact Extraction" and argues that verbatim-*grounded* extraction *wins* (CogCanvas 32.4% overall on LoCoMo vs RAG 24.6%; +20.6 pp on temporal reasoning, 32.7% vs 12.1%), whereas a later revision reportedly reverses to "Verbatim Chunks Beat Extracted Artifacts" (chunks 43.9% vs artifacts 28.0% on LoCoMo; 67.4% vs 45.4% on LongMemEval-S). The consistent, defensible takeaway regardless of version: **keep verbatim provenance, and let structured facts augment rather than replace raw text.** *Direct implication for AgentX:* it is extraction-first but *already stores verbatim turns with embeddings in `conversation_logs` and Neo4j `Turn` nodes.* It should make the episodic verbatim store a **first-class retrieval target in RecallLayer** (not just a consolidation input/audit mirror), keep `source_turn_id` provenance on every fact (it does), and preserve surface date strings during extraction. This is nearly free given the existing schema and is the single most important design takeaway for AgentX.

**Benchmark context (all cross-vendor numbers non-comparable — different judges, k, subsets, runs).** On LoCoMo (10 convs, ~600 turns/~26K tokens each, ~200 Q/conv; categories single-hop/multi-hop/temporal/open-domain; F1/BLEU/LLM-as-Judge): Mem0 J=66.9%, Mem0-graph J=68.4%, full-context J=72.9% (Chhikara et al., ECAI 2025, arXiv 2504.19413); Letta/MemGPT 74.0% with GPT-4o-mini (Letta, 2024 — cited in arXiv 2601.00821 §4.7 as an upper bound for stateful, database-backed systems). On LongMemEval (500 Q; five abilities: information extraction, multi-session reasoning, temporal reasoning, knowledge updates, abstention; long-context LLMs drop ~30 pts): per Emergent Mind's summary of Rasmussen et al. (Zep, 20 Jan 2025, arXiv 2501.13956), "Zep/Graphiti… achieving 71.2% (gpt-4o) overall accuracy with 2.6s latency versus 60.2% for vanilla full-context at 29s. Temporal-reasoning and multi-session QA see large relative improvements (17.3pp and 13.6pp)." Vendor 2026 self-reported (not independently replicated): Mem0's new algorithm LoCoMo 92.5 / LongMemEval 94.4 at <7K tokens/query; Graphiti LoCoMo 94.7% / LongMemEval 90.2%. **The Zep↔Mem0 LoCoMo dispute is unresolved:** per Atlan ("Zep vs Mem0," 2026), "Zep originally claimed 84%, Mem0 corrected this to 58.44%, and Zep counter-claimed 75.14%. Both vendors contest the other's methodology." AgentX should therefore build its own `eval_recall`/`eval_consolidation` harness (roadmap §2.7) rather than target published leaderboard numbers.

#### B.2 Learning over long periods with stable growth (Priority 2)

**Reinforcement-modulated forgetting — MemoryBank & the Ebbinghaus curve.** MemoryBank (Zhong et al., AAAI 2024) applies an Ebbinghaus-forgetting-curve update: frequently accessed, high-importance memories are reinforced while neglected ones fade, via a hand-set linear recency+relevance+importance score. AgentX's flat `×0.95^days` decay is weaker: it decays by time-since-access only, not modulated by *reinforcement strength*. *Mapping:* roadmap §2.4 (spaced-repetition decay: `stability` + `last_reinforced`, successful use extends half-life multiplicatively) is precisely the MemoryBank/Bjork upgrade. Bjork's New Theory of Disuse distinguishes **storage strength** (doesn't decay) from **retrieval strength** (decays adaptively) — argue for AgentX to model these as two fields rather than a single salience scalar.

**Don't delete — tombstone and compress.** The empirical caution (A-MEM ablations, reported in "Toward Efficient Agents"): forgetting-curve pruning effectively controls memory size and retrieval time but can cause "a substantial drop in task performance." Six formalized policies exist (FIFO, LRU, priority-decay, reflection-summary, random-drop, hybrid); the recommended production default is **LRU + priority decay + a salience floor** so high-importance facts can't be pruned on time alone, plus **reflection-summary (compress, don't delete)**. AgentX's `cleanup` job hard-deletes low-salience entities — conflicting with auditability and the event log. *Mapping:* roadmap §2.5 (unified tombstones: `retired_at`, recall-excluded, retention-swept; hard delete only on explicit user request) is the correct fix, to prioritize before any aggressive decay tuning.

**Summarization hierarchy — RAPTOR + graph community summaries.** RAPTOR (Sarthi et al., ICLR 2024, arXiv 2401.18059) recursively embeds→clusters (soft GMM)→summarizes chunks bottom-up into a tree, then retrieves across abstraction levels (collapsed-tree retrieval), improving multi-step-reasoning QA (e.g., +20% on QuALITY with GPT-4). This directly addresses "stable growth": summaries replace dozens of chunks, bounding token cost while preserving thematic recall. *Mapping:* AgentX has recap (conversation-level) and facts (shard-level) but **nothing in between** — roadmap §3.7 (Leiden community detection per channel → LLM-summarized `(:CommunitySummary)` nodes) fills this "altitude gap" and gives the Context Ledger's `shrink_fn` a principled degradation path (facts → community summary under pressure). RAPTOR is the document-side analogue; community summaries the graph-side analogue. Both are compatible (Neo4j GDS Leiden, or `networkx`).

**Consolidation as the core stability mechanism (CoALA / complementary learning systems).** CoALA (Sumers et al., 2023, arXiv 2309.02427) names episodic→semantic **consolidation** as the mechanism that makes the episodic/semantic pair productive; the neuroscience basis is Complementary Learning Systems (fast hippocampal store + slow neocortical consolidation). AgentX's 15-min consolidation sweep is exactly this pattern; the literature upgrades: (a) **event/idle-triggered** consolidation (roadmap §3.1) so the system isn't blind to its last 14 minutes; (b) **surprise-gated encoding** (roadmap §3.3) to concentrate compute where the situation changes; (c) **generative replay** (roadmap §3.9) — re-adjudicating contradiction-flagged/low-confidence clusters during idle time, i.e., "sleep-time compute" as offline policy improvement.

**Reflection & self-improvement — Reflexion + Generative Agents.** Reflexion (Shinn et al., NeurIPS 2023, arXiv 2303.11366) converts task feedback into first-person natural-language self-reflections stored in an episodic buffer; the paper states "self-reflection improves learning by an 8% absolute boost over the episodic memory learning advantage," and reports 91% pass@1 on HumanEval, surpassing the then-SOTA GPT-4 at 80%. Generative Agents' reflection tree synthesizes observations upward into durable "self-notions." *Mapping:* AgentX's reflex core is write-only; roadmap §3.8 (reflexion scoring — log a `procedure_violation` when a reflex procedure is in-prompt but a steer contradicts it; reinforce on steer-free matched-trigger turns) closes the loop and converts procedures into a measured control system. This is the procedural-memory analogue of Reflexion.

**Bitemporal knowledge & contradiction handling — Zep/Graphiti.** Graphiti (Rasmussen et al., 2025) annotates every edge with **event time** (when true in the world) and **ingestion time** (when observed), enabling non-lossy retroactive corrections and fact invalidation/supersession, with hybrid semantic+BM25+graph retrieval at P95 ≈ 300ms and no LLM calls at retrieval. *Mapping:* AgentX's single `temporal_context ∈ {current,past,future}` conflates *when-true* with *when-learned*; roadmap §1.7 replaces it with `valid_from/valid_to` + `asserted_at/superseded_at`, turning the 1.2×/0.7× multipliers into interval-overlap functions and making "what did we believe on March 1?" a query. This is the single biggest upgrade for temporal-reasoning accuracy (the category where all systems are weakest) and for stable handling of evolving facts.

**Agentic self-organizing memory — A-Mem.** A-Mem (Xu et al., 2025, arXiv 2502.12110) stores Zettelkasten-style atomic notes (content, keywords, tags, context, embedding, links); each new note triggers LLM-driven **link generation** to related notes and **memory evolution** (updating older notes' attributes), outperforming flat vector stores across six models. *Mapping:* AgentX's entity graph + `[:ABOUT]`/`RELATES_TO` edges + contradiction/supersession is a schema-constrained cousin. A-Mem's "memory evolution" is a caution as much as a technique: the roadmap's Invariant 4/6 (embeddings derived; CoT not persisted) and tombstone/audit discipline are exactly the "versioning/audit" safeguards A-Mem critics flag as missing. AgentX should *not* adopt silent note-rewriting without its bitemporal + tombstone substrate first.

#### B.3 Advanced continuations (Priority 3)

**Virtual context management — MemGPT/Letta.** MemGPT (Packer et al., 2023) treats context as OS-style paged virtual memory: main context (RAM) ↔ recall storage (searchable history) ↔ archival storage (vector), with the LLM paging via function calls (`archival_memory_search`, `core_memory_append`) and self-editing its own prompt; Letta adds a three-tier Core/Recall/Archival model and "sleeptime" background memory agents. *Mapping:* AgentX's Context Ledger is a **budget-allocator** analogue of MemGPT's paging, but AgentX does *not* currently give the LLM explicit self-paging tools or self-editing of the salient core. Two options: (a) keep the deterministic ledger (more auditable, fits AgentX's "glassbox" ethos) and add prefetch (roadmap §3.4); or (b) add MemGPT-style agent-controlled memory tools for a Tier-3 `deep_recall` specialist. The literature slightly favors (a) for reliability + the roadmap's Invariant 7 (hot-path recall never mints facts inline).

**Durable execution & checkpointing for resume.** Production practice (LangGraph checkpointers + Store; OpenAI Agents SDK sessions) separates **thread-scoped checkpoints** (conversation continuity, resume-after-crash) from **cross-thread stores** (long-term memory). Critical caution: on resume, nodes can re-execute, so external side effects need **idempotency keys**, and long-term facts need a `last_updated` field + a "verify preferences" flow when stale. *Mapping:* AgentX stores checkpoints/plan snapshots in Redis under `allkeys-lru` — a resumable plan can be silently evicted. Roadmap §1.6 (promote checkpoints/plan snapshots/rolling summary to PostgreSQL; split Redis into `volatile-lru` cache vs `noeviction` state) is the correct fix and a **prerequisite for reliable continuation**. The event-sourced write path (§1.1) makes L3 (transcript-only) resume lossless.

**Cross-session state via the salient core + goals.** AgentX's `get_salient_core` (maintained-not-searched high-salience block) + conversation-scoped goals + the Ambassador's cross-conversation survey are a strong continuation substrate — closer to LangGraph's "Store" than most systems. The upgrade is to make the salient core *bitemporally correct* (exclude superseded facts by validity interval, §1.7) and *reinforcement-weighted* (§2.2/§2.4) so resumed sessions surface durable, current knowledge.

| C9 | Reflex core is write-only | `memory/procedural.py`, `streaming/steering.py` | Reflexion scoring / procedure-violation signal (§3.8) | Medium |
| C10 | Resumable state under Redis LRU eviction | `connections.py`, checkpoint/plan storage | Promote to PostgreSQL; Redis durability tiers (§1.6) | High (continuation) |
| C11 | Dual-write drift (Neo4j vs PG) | `store_turn`, `_persist_turns` | Event-sourced `conversation_logs` write-ahead log + `rebuild_memory_projections` (§1.1) | High (continuation + durability) |
| C12 | Static confidence calibration | `extraction/service.py` | Isotonic calibration fitted per `extraction_model` (§2.3) | Low–Medium |
| C13 | No retrieval eval harness | new `eval_recall` | Golden set (query→expected-fact-ids), recall@k/MRR per technique | **Prerequisite for C1/C2/C4** |

## Recommendations

**Stage 0 — Instrument before optimizing (do first).** Build the `eval_recall` golden-set harness (C13/§2.7): a seeded corpus with (query → expected-fact-id) pairs, reporting recall@k and MRR per technique and fused, on the existing snapshot/restore utility. Add per-item **score-decomposition logging** (vector vs salience vs recency vs temporal) to `memory_audit_log` (§2.2). *Rationale:* every accuracy change below is otherwise invisible, and the cross-vendor benchmark disputes prove you cannot trust external numbers. **Threshold to proceed:** a stable baseline recall@10/MRR on ≥50 golden queries.

**Stage 1 — Accuracy quick wins (highest ROI, stack-native).**
1. **Cross-encoder reranking (C1):** insert a `bge-reranker` cross-encoder (available via `sentence-transformers`, already a dependency) as a second stage over ≥50 hybrid-RRF candidates → top-8. Largest documented lever (+12–17 pp Recall/MRR in the literature). **Benchmark to keep it:** ≥+5 pp MRR on the Stage-0 golden set; drop if the retrieval P95 latency budget (target < ~500 ms, cf. Zep's ~300 ms) is exceeded without accuracy gain.
2. **Promote verbatim turns to first-class recall (C3)** and preserve surface date strings in extraction prompts. Near-zero cost given `conversation_logs` already holds embedded turns.
3. **Personalized PageRank over the entity graph (C4)** behind `recall.ppr_enabled`, via Neo4j GDS (fallback bounded traversal / `networkx`), with node-specificity weighting. **Threshold:** measurable multi-hop recall gain on golden queries tagged multi-hop.

**Stage 2 — Structural stability (bounded growth + correct time).**
4. **Bitemporal facts (C5)** + **unified tombstones (C6)** together — they share the contradiction/supersession pipeline and jointly fix temporal accuracy *and* stable growth. Ship tombstones first (stop hard-deleting), then the additive bitemporal migration (provenance-stamped, per §1.7's resolved plan).
5. **Reinforcement-modulated decay (C7):** replace flat `0.95^days` with `stability`/`last_reinforced`, incrementing reinforcement only when a fact demonstrably mattered (survived ledger allocation / was cited), and reserve one top-k slot as an exploration floor to break the salience feedback loop.

**Stage 3 — Frontier, eval-gated (each flag-gated, guarded by the Stage-0 harness).**
6. **Adaptive dispatch (C2):** ship a rule-based router first (route by query shape/intent/channel), then upgrade to the §3.5 contextual bandit once the usage-reward signal is logged.
7. **Summarization hierarchy (C8):** Leiden community summaries + RAPTOR-style recursive summaries, wired into the Context Ledger's `shrink_fn` as the principled degradation altitude.
8. **Reflexion scoring (C9)** and **generative replay (§3.9)** to close the procedural loop and re-adjudicate contested facts during idle time.

**Continuation track (parallel, prioritize C10/C11 early).** Move checkpoints/plan snapshots/rolling summaries out of LRU-evictable Redis into PostgreSQL (C10), and land the event-sourced write path (C11) so resume is lossless (L3 transcript-only degradation). Add idempotency keys to any tool call replayed on resume. Continuation-critical and independent of the accuracy work.

**What would change these recommendations:** if the Stage-0 harness shows cross-encoder gains < 5 pp on AgentX's actual corpus, deprioritize C1 in favor of C3/C4 (representation + graph). If temporal reasoning is not a target use case, defer C5's bitemporal migration (the most invasive change). If AgentX moves multi-user/multi-cluster, C11 (event-sourced log) jumps to highest priority because dual-write drift becomes a correctness bug, not just a durability one.

## Caveats

- **Repo access limitations (flagged for the next iteration):** This report is based on the public docs site (`/docs/features/memory`, `/docs/architecture/memory`, `/docs/architecture/memory-capabilities`), the `Readme.md`, `pyproject.toml`, and the 525-line `Memory-Roadmap.md`. **I did not read the actual source files** (`memory/recall.py`, `recall/layer.py`, `extraction/service.py`, `consolidation/jobs.py`, `agent/context_ledger.py`, the `todo/backlog/*.md` files, `Decisions.md`, `Development-Notes.md`, `Repo-Questions.md`). Implementation details (exact RRF k, whether a cross-encoder is already present, whether verbatim turns are already a recall target, current bandit/PPR status) may differ from the docs. The `uv.lock` was not parsed line-by-line; dependency facts come from `pyproject.toml`. Docs cite Neo4j server "5.15+" while `pyproject.toml` pins driver `neo4j>=6.0.3` — verify the deployed server version.
- **Version skew:** `Memory-Roadmap.md` is reconciled to v0.21.88; the live site reports v0.21.147. Items marked `[SHIPPED]`/`[PARTIAL]` (e.g., Context Ledger §1.8) may have advanced. Several roadmap items cited here as "planned" may already be partly shipped.
- **The verbatim-vs-extracted finding (arXiv 2601.00821) is contested and its exact figures are unverified:** it is a single-author, non-peer-reviewed preprint that appears to exist in conflicting versions with *opposite* conclusions and titles ("CogCanvas: Verbatim-Grounded Artifact Extraction" arguing extraction wins at 32.4% vs 24.6% on LoCoMo, vs a reported later revision "Verbatim Chunks Beat Extracted Artifacts" arguing chunks win 43.9% vs 28.0%). Treat it as a strong prompt to *retain verbatim provenance and preserve surface detail*, **not** as proof that extraction is worthless — AgentX's extraction also powers graph reasoning that pure chunks cannot support. The robust, version-independent conclusion is "structured memory should augment, not replace, verbatim text."
- **Benchmark numbers are largely non-comparable across systems** (different judges, k-values, category subsets, single vs averaged runs, backbone models); the Zep↔Mem0 LoCoMo dispute is explicitly unresolved (Zep 84% claim → Mem0-corrected 58.44% → Zep counter-claim 75.14%), and 2026 vendor numbers (Mem0 92.5/94.4, Graphiti 94.7/90.2) are self-reported and not independently replicated. Use them only as rough context; rely on AgentX's own harness.
- **Speculative/forward-looking content** in the roadmap uses "planned"/"will" framing and should not be read as shipped. Home-page marketing copy ("memory that actually learns," "context that never overflows") is aspirational.
